# SigV4 Authentication — Design & Implementation Plan

## Overview

Add AWS Signature Version 4 request authentication to within.website services.
SigV4 is HMAC-SHA256 based: a client signs a request with a secret access key,
and a verifier recomputes the signature and compares it in constant time. This
covers both directions — verifying incoming signed requests and signing outgoing
ones — and the IAM machinery (users, keys, a key generator) that backs them.

The central design question was **where signature verification runs**, which
decides where signing secrets live. That choice drove the whole component shape.

## The architecture decision: local vs. central validation

SigV4 is symmetric (HMAC), so verification always needs the secret _somewhere_ —
there is no public-key shortcut. Two models:

- **Local validation** — the verifying service holds (or fetches) the secret and
  recomputes the signature itself. Fast, no per-request RPC, but every
  verifying service is a secret custodian: a compromised one can exfiltrate
  long-lived secrets usable against every service.
- **Central validation** — the verifier forwards the received request material
  to a trusted STS service (here, `iamd`), which validates and returns the
  caller identity. Secrets never leave STS; a compromised verifier can only
  forge auth _through_ STS while its credential is live, and rotating that
  credential locks the attacker out instantly.

**Decision:** central validation is the default for downstream services that
must not be trusted with the key store. Local verification stays available for
the case where the verifier is trusted with secrets (or for testing). The
real cost of central validation is **availability coupling** (iamd becomes a
synchronous auth dependency), not latency — mitigated by running iamd HA and
co-located, and failing closed (503) on iamd outages.

### Addendum (2026-07-06): from per-request central validation to derived-key caching

The decision above shipped as `STSService.GetCallerIdentity`: every verifier
forwarded the full request material to iamd on every call. Running it for a
while surfaced the real cost more sharply than expected — it put an RPC on
the hot path of _every_ protected request, not just cold ones, and it made
iamd a synchronous auth dependency for every downstream service under load,
not merely during outages.

The decision is revised: verifiers now fetch a derived SigV4 signing key from
`SigningKeyService.GetSigningKey` — `HMAC(HMAC(HMAC(HMAC("AWS4"+secret,
date), region), service), "aws4_request")` — scope-pinned to one `(access_key
id, UTC day, region, service)` tuple, with a `±1` day request-date window and
a hard `not_valid_after` at the scope day's end plus 15 minutes of clock
skew. `iamsts` caches the key until `cache_until` (`min(now +
-signing-key-cache-ttl, not_valid_after)`, default TTL 5 minutes) and
verifies signatures locally against it. This keeps the original property that
motivated central validation — a verifier never holds a raw secret access
key — while removing iamd from the request hot path entirely once a scope's
key is cached. The trade made explicit: a leaked response now exposes secret
material bounded to one scope (one key, one day, one region, one service)
rather than nothing, and revocation is no longer instant — it is bounded by
`-signing-key-cache-ttl` instead of by iamd's next response. `GetCallerIdentity`
and the per-request central-validation path it required have been deleted.
One more trade-off this surfaced: any authenticated IAM principal may
currently fetch any principal's derived signing key for the fleet's
`(region, service)` scope within the `±1` day window — `GetSigningKey` does
not check that the caller and the requested `access_key_id` belong to the
same principal. Every fetch is attributed to the caller in logs/metrics, so
misuse is visible after the fact even though it is not prevented up front. A
verifier allowlist (restricting who may request whose key) is a follow-up
decision if end-user and service credentials ever become distinct trust
tiers.

## Components

| Component        | Path                                       | Role                                                                                    |
| ---------------- | ------------------------------------------ | --------------------------------------------------------------------------------------- |
| Local verifier   | `web/middleware/sigv4`                     | Recomputes SigV4 signature with a `Lookuper`-supplied secret                            |
| Key generator    | `web/middleware/sigv4/sigv4keygen`         | `crypto/rand` access/secret key pairs                                                   |
| Outgoing signer  | `web/middleware/sigv4/sigv4client`         | `http.RoundTripper` that signs outgoing requests (vendored Prometheus, Apache-2.0)      |
| Central verifier | `web/middleware/sigv4/iamsts`              | Fetches derived signing keys from SigningKeyService, caches per scope, verifies locally |
| STS contract     | `pb/within/website/x/iam/sts/v1/sts.proto` | `GetSigningKey` RPC                                                                     |
| IAM service      | `cmd/iamd`                                 | Users, keys, and the signing-key distribution server                                    |

## Security model — division of responsibility

_Superseded 2026-07-06 by the derived-signing-key model described in the
addendum above; kept below for history._

- **STS (`iamd`)** verifies the signature, the credential scope, the clock
  window, and that the key/user are enabled. It confirms the signature covers
  the declared payload hash and the signed headers.
- **The verifying service** confirms the bytes it actually received hash to the
  declared `x-amz-content-sha256`. STS never sees the body, so this check must
  run where the body is — otherwise a body swapped after signing goes
  undetected.
- **mTLS** between the verifier and STS, so STS can authenticate the verifier
  and the channel carrying the forwarded headers is trusted.

## Hardening plan (the code review)

A review of the original `sigv4` package surfaced 13 defects; scope covered all
of them (replay-within-skew, #14, is inherent to bare SigV4 and out of scope).

| #   | Finding                                                     | Fix                                         |
| --- | ----------------------------------------------------------- | ------------------------------------------- |
| 1   | `sigv4keygen` secret was `HMAC(ak, time)` — predictable     | `crypto/rand` for both keys                 |
| 2   | Streaming payloads accepted with no chunk verification      | Reject `STREAMING-…-PAYLOAD`                |
| 3   | `sigv4client` mutated the caller's request (retry hazard)   | Clone first, mutate only the clone          |
| 4   | `host` not required in `SignedHeaders` (cross-vhost replay) | Require `host`                              |
| 5   | `MaxBodySize` bypassed for unsigned/streaming payloads      | Enforce in `readAndLimitBody` for all paths |
| 6   | `Date`-header fallback was dead code                        | Real fallback via `http.ParseTime`          |
| 7   | Non-sentinel errors mapped to 403, masking outages          | `default` arm → 500                         |
| 8   | `Verifier.Lookup == nil` panicked                           | Guard → `ErrNotConfigured`                  |
| 9   | `sigv4client` signed with empty region from default chain   | `region: awscfg.Region`                     |
| 10  | Body-hash compare case-insensitive but canonical used raw   | Return lowercase `computed`                 |
| 11  | `collapseSpaces` ignored quoted strings its doc promised    | Quote-aware folding                         |
| 12  | `parseAuthHeader` silently dropped malformed components     | Hard-fail on bad/unknown `k=v`              |
| 13  | Body not reset on error paths                               | `readAndLimitBody` always resets            |

## Decisions locked

- **Streaming → reject** (chunk-signature verification is out of scope; rejecting
  is safe and honest).
- **Patch the vendored `sigv4client`** in place; keep the Apache-2.0 header.
- **Central validation** as the default verification model for untrusted
  downstream services.
- **`DisableKey` enforces `user_id` scope** — a caller scoped to one user cannot
  revoke another user's key (returns not-found, no existence leak).

## Status

**Done:**

- `sigv4` local verifier hardened, with SDK round-trip + tamper/skew/scope tests.
- `sigv4keygen` using `crypto/rand`.
- `sigv4client` request-mutation and region bugs fixed.
- `sts.proto` (`GetCallerIdentity`) defined, generated (gRPC + Twirp).
- `iamsts` central-validation middleware with Twirp client + body-hash check +
  error mapping, tested against a stub STS over a real Twirp round-trip.
- `iamd` keys service (`CreateKey`/`DisableKey`/`ListKeys`), wired into `main.go`.
- `iamd` DAO: `GetUser`, `GetUserByAccessKeyID`, scoped `DisableKey`,
  `SecretFor` (verifier `Lookuper`), `Open` (test-friendly, no Prometheus).
- Table-driven tests for the DAO and the keys service (incl. scope enforcement).
- `sigv4` exports `VerifySignature` — bodyless signature verification for
  central STS validation: it verifies against the declared payload hash without
  the body (the verifying service checks the body locally), reusing the hardened
  canonicalization core. Streaming payloads are rejected.
- `iamd` hosts the STS **server** (`cmd/iamd/services/iam/sts`):
  `GetCallerIdentity` reconstructs the request, verifies with `VerifySignature`,
  maps verification failures → `twirp.Unauthenticated`, resolves the owning user,
  and returns it. Covered by table-driven and end-to-end integration tests.
- Every `iamd` route (`UserService`, `KeyService`, `STSService`) is wrapped in
  the local `sigv4.Verifier` middleware, so every call requires a valid
  signature — **including the STS route**. This replaces the mTLS plan: a
  downstream verifier authenticates to STS by signing its `GetCallerIdentity`
  Twirp call with its own IAM credential, verified by the same middleware.
- `iamd` bootstrap (`-bootstrap-username`): creates the first admin user + key
  in-process when the DB is empty; the secret is generated and logged once
  (never accepted via flag/env); idempotent.
- `iamsts` fixes: forwards `content-length` (the AWS signer signs it from the
  `ContentLength` field, which net/http strips from the header map, so it must
  be carried explicitly for body-bearing requests), and maps `ErrBodyHash` /
  `ErrBodyTooLarge` to `twirp.PermissionDenied` / `InvalidArgument` (fixes a
  forgery that was masked as a 500).

**Done (2026-07-06 revision — derived signing-key caching, see addendum above):**

- `sigv4` exports `DeriveSigningKey` and `SigningKeyLookuper` (context-aware
  `LookupSigningKey(ctx, accessKeyID, date, region, service)`), so a verifier
  can validate a signature against a derived key instead of a raw secret.
- `pb/within/website/x/iam/sts/v1/sts.proto` replaces `STSService` /
  `GetCallerIdentity` with `SigningKeyService.GetSigningKey`, scope-pinned
  request validation (`buf.validate` on `access_key_id`/`date`/`region`/
  `service`), and a response carrying `signing_key`, `identity`,
  `not_valid_after`, and `cache_until`.
- `iamd` DAO: `GetKeyWithUser` distinguishes an unknown access key id from a
  disabled one (`NOT_FOUND` vs. `PERMISSION_DENIED`), needed so
  `GetSigningKey` can report the right error without leaking which case it is
  to an unauthenticated-looking probe.
- `iamd` hosts `SigningKeyService` (`cmd/iamd/services/iam/sts`):
  `GetSigningKey` validates the requested scope is this deployment's
  `(region, service)` and within a `±1` day window of the credential date,
  derives the key, and sets `not_valid_after` to the scope day's end plus 15
  minutes of clock skew and `cache_until` to `min(now +
-signing-key-cache-ttl, not_valid_after)`. Neither the raw secret nor the
  derived key is logged. Covered by table-driven tests.
- `iamsts` rewritten as a caching local verifier (`iamsts.New(iamsts.Config{...})`):
  exact-`(access_key_id, date, region, service)`-tuple cache, singleflighted
  fetches so concurrent misses for one scope collapse into one RPC, a 30s
  negative cache for refused credentials, and expired-scope sweeping on
  insert. Verified identity is exposed via `iamsts.Caller(ctx)` — an
  `*iamsts.Identity` with `PrincipalID` set to iamd's user UUID.
- `cmd/iamd` end-to-end integration test covers the full chain: bootstrap →
  downstream verifier fetches a derived key → end-user request verified
  locally with zero further RPCs while warm → a disabled key stops verifying
  once its cached entry expires.
- The per-request central-validation flow is fully deleted: `STSService`,
  `GetCallerIdentity`, and `sigv4.VerifySignature` (the bodyless verification
  helper it depended on) no longer exist anywhere in the tree.
- `iamd` exposes `-signing-key-cache-ttl` (default 5m), the fleet-wide bound
  on how long a downstream verifier may cache a derived key before
  re-fetching — the ceiling on revocation latency.

**Remaining:**

- None. The derived-key-caching replacement described in the addendum above
  has shipped, the legacy central-validation flow it replaces has been
  deleted, and all of the above is committed on the `Xe/sigv4` branch.

## Wiring & usage

`iamd` runs as both the IAM service and the central STS. Start it on a fresh DB
and capture the bootstrap credentials from the logs:

```sh
iamd -db-loc ./var/iamd.db -bootstrap-username root
# logs: access_key_id=… secret_access_key=…
```

Region/service are **fleet-wide** (`-region`, `-service`, defaults `us-east-1` /
`iam`); every client and every downstream verifier must sign with the same scope.

A downstream service that must not hold secrets validates incoming requests via
`web/middleware/sigv4/iamsts`, pointing its `SigningKeyService` client at iamd
and signing each `GetSigningKey` call with the verifier's own IAM credential
through `sigv4client`. The verifier caches derived keys per scope, so warm
traffic never calls iamd:

```go
signedRT, _ := sigv4client.NewSigV4RoundTripper(&sigv4client.Config{
    Region: "us-east-1", AccessKey: verifierAKID, SecretKey: verifierSecret,
    ServiceName: "iam",
}, nil)
verifier := iamsts.New(iamsts.Config{
    BaseURL:     "https://iamd.example",
    HTTPClient:  &http.Client{Transport: signedRT},
    Region:      "us-east-1",
    Service:     "iam",
    MaxBodySize: 1 << 20,
})
http.ListenAndServe(":8080", verifier.Middleware(handler))
```

Downstream handlers read the verified caller via `iamsts.Caller(ctx)`, which
returns an `*iamsts.Identity` with `PrincipalID` set to iamd's user UUID.
`iamd` itself controls how long a derived key may be cached via
`-signing-key-cache-ttl` (default 5m) — the ceiling on revocation latency
after a key is disabled.

For services trusted with the key store (iamd itself, or tests), use the local
`web/middleware/sigv4.Verifier` directly with a `Lookuper` over `SecretFor`.
